# Задача: сделать grace-board диспетчер полностью автономным

> Открой новую сессию Claude Code **в проекте `~/Projects/grace-board`** и выполни эту задачу.

Ты работаешь в проекте `~/Projects/grace-board` — это локальный kanban-диспетчер для
пайплайна `/grace-feature-dev`. Вся логика в одном файле: `server.js` (~705 строк).
Node, без сборки. Запуск: `node server.js`. Состояние: `data/board.json` (карточки
этапов). Каждая карточка порождает pipeline-доску в целевом проекте:
`<project>/.grace-feature-dev/<slug>/board.json`.

ПЕРЕД началом прочитай память с разбором бага:
`~/.claude/projects/-Users-vm-Projects-grace-board/memory/grace-board-dispatcher-build-gap.md`

---

## Желаемое поведение (definition of done)

Человек перетаскивает карточку из Backlog → дальше ВСЁ происходит само, без ручных
правок файлов:

1. spawn ask-functional → решает, есть ли что спрашивать у человека.
2. если есть функциональные вопросы → пишет `questions`, `column:"asking"`, ждёт человека.
3. spawn ask-architecture → решает, нужны ли развилки.
4. **РАЗВИЛКИ ЕСТЬ:** пишет `archQuestions`, ждёт выбора человека → человек выбирает →
   запускается ПОЛНЫЙ build до `ready`.
5. **РАЗВИЛОК НЕТ:** диспетчер САМ запускает ПОЛНЫЙ build до `ready` — без участия
   человека и без ложного `blocked`.
6. **ВОПРОСОВ К ЧЕЛОВЕКУ ВООБЩЕ НЕТ** (типично для core/infra-этапов): обе фазы вопросов
   пустые → этап проходит насквозь до `ready` полностью автономно.
7. Карточка едет `implementing→verifying→reviewing→ready` сама; per-card колонки и
   `verdict` обновляются; доска никогда не врёт.

## Корень проблемы (два бага)

**Баг 1 — путь «развилок нет» не идёт через `launchBuild`.**
В `launchAskArchitecture` (≈ строка 280) ветка «ЕСЛИ развилок нет» инлайн-инструктирует
тот же ask-процесс «продолжай ПОЛНЫЙ build» одной строчкой-заглушкой. У неё НЕТ полных
директив из `launchBuild` (≈ строка 317): обязательного board-sync в начале каждой фазы,
per-card статусов, рамки «гони до ready, не останавливайся». Агент делает частичный слой,
пишет «готов продолжить по команде» и выходит → процесс мёртв → watchdog (≈ строки
378–390, `isAlive` + `ACTIVE_COLUMNS`) видит «pid мёртв + станция implementing» и ставит
`blocked`. Полноценный `launchBuild` вызывается ТОЛЬКО из ветки architecture в обработчике
`/answers` (≈ строка 537) и из relaunch-с-archDecisions.

**Баг 2 — relaunch не доходит до build для этапов без развилок.**
В `/relaunch` (≈ строки 616–624) точка входа выбирается так:
`archDecisions.length → build; иначе answers.length → ask-architecture; иначе → functional`.
Для этапа без развилок `archDecisions` пуст, но `answers` есть → relaunch снова уходит в
ask-architecture и повторяет ту же поломку. Пути к build без архитектурных решений нет.

## Что сделать (логика автономности)

1. **Развести ответственность `launchAskArchitecture`:** пусть он ТОЛЬКО классифицирует и
   выходит, НИКОГДА не строит инлайн.
   - развилки есть → `archQuestions:[...]`, `askStage:"architecture"`, `column:"asking"`, выход;
   - развилок нет → `archQuestions:[]`, `archDecisions:[]` (пустой массив = маркер «гейт
     пройден, выбирать нечего»), `askStage:"done"`, `column:"asking"`, выход.

2. **Авто-запуск настоящего build.** Добавь в супервизор `syncFromPipeline` (цикл ≈ 348–392)
   детект состояния «гейт архитектуры пройден, развилок не было, build ещё не стартовал» →
   вызови `launchBuild(...)` (единый источник build-промпта), выставь `column:"implementing"`,
   `recordLaunch(card, launch, "build")`, запиши событие в `DISPATCH_LOG`.
   - Условие примерно: `pip.askStage === "done"` И `(pip.archQuestions||[]).length === 0` И
     `card.runKind !== "build"` (или отдельный флаг `card.buildLaunched`).
   - ОБЯЗАТЕЛЬНО защити от повторного спавна (идемпотентность: один build на карточку).
     После авто-запуска watchdog должен сторожить НОВЫЙ build-pid, а не мёртвый ask-pid.

3. **Почини relaunch** (≈ 616–624): различай «archDecisions пуст, потому что развилок не
   было (гейт done)» и «архитектура ещё не пройдена». Новая логика выбора входа:
   - `archDecisions.length` (выбраны развилки) → build;
   - иначе `askStage==="done"` (гейт пройден, в т.ч. no-fork с `archDecisions:[]`) и есть
     `answers` → build;
   - иначе есть `answers` (функционал пройден, архитектура нет) → ask-architecture;
   - иначе → ask-functional.

## Калибровка «высоты» вопросов (КЛЮЧЕВОЕ)

Проблема на практике (этапы 0.1/0.2): пайплайн задавал человеку слишком ДЕТАЛЬНЫЕ,
системно-имплементационные, почти атрибутные вопросы (где ставить SET LOCAL, имена
PG-ролей и GUC, mixin vs Alembic-хелпер, глубина тестов). По ним у заказчика нет
выбора — это не требования и не продуктовые/архитектурные развилки. Их нельзя
эскалировать человеку: он физически не может их решить.

Введи жёсткий порог эскалации в промптах `launchAskFunctional` (≈258) и
`launchAskArchitecture` (≈280):

**Спрашивать человека МОЖНО только если решение:**
- меняет ПОВЕДЕНИЕ продукта или объём (что система делает/не делает для пользователя), ЛИБО
- это настоящая архитектурная развилка с внешними последствиями: стоимость, vendor
  lock-in, юридка/комплаенс (GDPR/резидентность/провайдер), необратимость, форма модели
  данных, влияющая на контракт, ЛИБО
- по нему у НЕ-разработчика (продукт/юрист/владелец) реально может быть мнение.

**НЕ спрашивать (решать самому, по умолчанию):**
- расположение кода, имена ролей/переменных/GUC, паттерн (mixin vs helper), где ставится
  SET LOCAL, структура файлов, глубина/способ тестирования, формат логов и т.п. —
  всё системное/имплементационное/«как», а не «что».

**Поведение по умолчанию:** агент решает такие вопросы САМ, по best-practice и инвариантам
проекта (CLAUDE.md / ARCHITECTURE.md), и ЗАПИСЫВАЕТ принятое решение с кратким обоснованием
в `board.json` (functional → `answers:[{q,a}]`; architecture → `archDecisions:[{q,chosenTitle,
ownText(rationale)}]`) — чтобы был след, но БЕЗ блокировки на человеке.

**Калибровка по типу этапа:** для фундаментальных core/infra-этапов (как 0.1/0.2) ожидаемое
число вопросов к человеку — как правило 0. Тогда:
- `launchAskFunctional`: если человеку спрашивать нечего — пиши `questions:[]`, сам заполни
  `answers` принятыми решениями, `askStage:"functional-done"` (или эквивалент), и НЕ
  останавливайся на человеке.
- `launchAskArchitecture`: `archQuestions:[]` + `archDecisions:[]`/самостоятельные решения,
  `askStage:"done"` → срабатывает авто-launchBuild из правки выше.

Иными словами: «asking» — это не «вывали всё подряд», а «есть ли здесь хоть один настоящий
выбор для человека». Если нет — этап проходит насквозь до build автономно.

**Связанная правка сервера:** обработчик `/answers` и супервизор должны корректно
обрабатывать `questions:[]` (нечего спрашивать) — не зависать в `asking`, а двигаться
к следующему гейту / в build. Когда functional не задал вопросов, диспетчер должен сам
запустить ask-architecture (а не ждать `/answers` от человека).

## Ограничения

- НЕ ломай рабочий путь «развилки есть → человек выбирает → build» (`/answers` architecture).
- `column` всегда из словаря `COLUMNS` (строка 58); `done`/`clarifying` нормализуются через
  `normalizeColumn` (61–62) — не вводи значений вне словаря.
- `board.json` (и проектная, и pipeline) — единственный источник истины; не держи состояние
  в памяти между тиками (супервизор читает `readBoard()` каждый тик — сохрани это свойство).
- Маленькие точечные правки; не переписывай файл целиком. Объясни каждое изменение.

## Проверка (обязательно прогнать)

- Синтаксис: `node --check server.js`.
- Сценарий A (без развилок): эмулируй карточку, у которой ask-architecture пишет
  `archQuestions:[]`+`askStage:"done"`. Убедись, что супервизор САМ зовёт `launchBuild`
  один раз, карточка идёт в `implementing`, ложного `blocked` нет.
- Сценарий B (с развилками): путь через выбор человека по-прежнему ведёт в build.
- Сценарий C: relaunch упавшего no-fork build возобновляет build, НЕ ask-architecture.
- Сценарий D (ноль вопросов): этап, где обе фазы вопросов пустые, доходит до `ready`
  вообще без участия человека.
- Анти-дубль: build-процесс не спавнится дважды.
- Используй `GRACE_AUTORUN=0`, чтобы `spawnRun` не дёргал реальный `claude` (см. строку 241),
  и/или каталог `dryrun_gfd/` для прогона без живых ран — проверяй переходы состояния доски,
  а не саму сборку.
