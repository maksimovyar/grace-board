# Промт: доработка LA4 (самоисцеление) — две дыры

> Тема для карточки grace-board / аргумент к `/grace-feature-dev`.
> Проекты, которые трогаем: **grace-board** (`server.js`) и **grace-lib**
> (`~/.claude-libs/grace/` — команда + SKILL). Обе директории добавить в `--add-dir`.

---

Доработай контур самоисцеления (LA4) пайплайна grace-feature-dev. Сейчас LA4 = build-loop
с Anti-Loop (verify/review fail → карточка назад в `implementing`, при `attempts ≥
antiLoop.max` → `blocked`). Две конкретные дыры, закрой обе:

## Дыра 1 — нет пер-карточных green-чекпоинтов (нечего откатывать)

Код коммитится ровно один раз — в конце всей фичи: run работает в ветке `autodev/<slug>`
и делает `git push` только перед `column:"ready"` (см. build-launch промт в
`grace-board/server.js`, блок BRANCH & HANDOFF, ~строки 355-359, функция `launchBuild`).
Внутри сборки код не чекпоинтится по карточкам: если карточка N портит файл, затронутый
карточкой N-3, откатываться не к чему — `board.json` хранит состояние процесса, но не код.

**Сделай:** каждый раз, когда карточка проходит verify+review зелёными и переходит в
`done`, run делает **микро-коммит на ветке `autodev/<slug>`**:
`git add <только files[] этой карточки> && git commit -m "green(<cardId>): <краткий title>"`.
Это и есть LA4 «вечно зелёный билд» + точки отката.

- Коммитим **только** файлы из `card.files[]` (не `git add .`) — чтобы чекпоинт карточки
  был атомарным и не тянул чужие изменения.
- Только на зелёной карточке (verify + review прошли). Провал → НЕ коммитим.
- Финальный `push` перед `ready` остаётся как есть — эти green-коммиты в ту же ветку.
- При провале Anti-Loop, когда карточка уходит в `blocked`, оставляем последний зелёный
  чекпоинт нетронутым (не откатываем сборку целиком) — человек продолжит от него.

**Где править (это инструкции run'а, а не рантайм-код):**
- `grace-board/server.js` → функция `launchBuild`, промт-строка: в блок про карточки
  («PER-CARD STATUS…» / «DEFINITION OF DONE») добавь директиву green-checkpoint-коммита
  при переходе карточки в `done`.
- `~/.claude/commands/grace-feature-dev.md` → Phase 5-6, шаг 5 «Resolve» (там где «move card
  to `done`, record verdict, journal it») — добавь green-commit как явный шаг.
- `~/.claude-libs/grace/skills/grace-feature-dev/SKILL.md` → раздел про переход карточки в
  `done` (card lifecycle) — зафиксируй формат сообщения `green(<cardId>): …` и правило
  «только card.files[]».

## Дыра 2 — восстановление после смерти/зависания run'а только ручное

Сторож в `grace-board/server.js` (watchdog, ~строки 483-492 + `blockCard` ~377): если pid
умер, не достигнув `ready` → сразу `blockCard`; если станция не менялась > `STALL_MS`
(`GRACE_STALL_MIN`, деф. в env) → `SIGTERM` + `blockCard`. Дальше человек жмёт ↻ relaunch
(endpoint `POST /api/tasks/:id/relaunch`, ~строка 708). То есть на каждый упавший прогон —
ручное вмешательство, даже если причина тривиальная.

**Сделай:** перед тем как звать человека, сторож пытается **авто-исцелиться один раз**:

1. На первом событии смерти/зависания в активной станции: НЕ блокировать сразу. Считать
   хвост лога прогона (последние ~150 строк того же лог-файла, что показывает `⊟ log`).
2. **Авто-relaunch один раз**, переиспользуя существующую resume-логику relaunch-эндпоинта
   (тот же выбор точки возобновления по `askStage`/`archDecisions`). В resume-промт
   **инжектнуть RECOVERY-контекст**: «Предыдущий прогон УМЕР/ЗАВИС на станции `<column>`.
   Хвост лога: <tail>. Диагностируй причину и продолжи с самого дальнего зелёного
   чекпоинта.» (Диагностирует сам возобновлённый run по логу — отдельный вызов модели НЕ
   нужен, лишнюю квоту не жжём.)
3. Пометить, что авто-исцеление уже было: инкремент `card.autoHealCount` (новое поле).
   Записать событие `autoheal` в `data/dispatch-log.ndjson` (по образцу события `relaunch`).
4. **Только если** `autoHealCount >= 1` (второй провал) ИЛИ авто-relaunch сам упал —
   тогда `blockCard` как сейчас, но в `blockReason` добавить «авто-исцеление не помогло».

**Где править:** `grace-board/server.js` — watchdog-цикл и `blockCard`/relaunch. Вынеси общую
resume-логику relaunch в функцию, чтобы watchdog её переиспользовал (сейчас она инлайн в
обработчике `POST …/relaunch`). Не дублируй spawn-код.

## Ограничения (соблюдать — из глобального CLAUDE.md)

- Никаких секретов в коде; ничего не коммитить в `main` напрямую (green-коммиты только в
  `autodev/<slug>`).
- `git add` строго по `card.files[]` — не `git add .` и не `-A`.
- Авто-исцеление строго **один раз** на карточку/прогон — никаких циклов relaunch (это бы
  сломало смысл Anti-Loop и жгло квоту). `autoHealCount` — жёсткий предохранитель.
- Сервер без зависимостей (plain Node ≥18), bound на `127.0.0.1`, CSRF-guard не трогать.
  Авто-relaunch инициирует сам сервер (не внешний запрос) — CSRF к нему не применяется, но
  проверку `isInsideRoot(projectDir)` перед spawn сохрани.
- `board.json` остаётся единственным источником состояния доски; новые поля (`autoHealCount`)
  — аддитивно, backward-compatible со старыми board.json (отсутствие = 0).

## Критерии приёмки (измеримые)

1. **Green-checkpoint:** после прогона фичи из ≥2 карточек `git log --oneline` в ветке
   `autodev/<slug>` содержит по коммиту `green(<cardId>): …` на каждую зелёную карточку +
   финальный. `git show --stat` каждого green-коммита показывает ТОЛЬКО файлы этой карточки.
2. **Откат работает:** после порчи файла на поздней карточке `git restore --source=green(<id>)`
   возвращает к зелёному состоянию соответствующей карточки без потери остальных.
3. **Авто-heal:** искусственно убитый прогон (kill pid до `ready`) → карточка НЕ уходит сразу
   в `blocked`; в `dispatch-log.ndjson` появляется событие `autoheal`; `card.autoHealCount==1`;
   run перезапущен с RECOVERY-контекстом в промте.
4. **Предохранитель:** второй подряд провал того же прогона → карточка → `blocked` с
   `blockReason`, упоминающим неудачное авто-исцеление. Третьего авто-relaunch НЕ происходит.
5. Старые `board.json` без `autoHealCount` читаются и работают (трактуются как 0).

## Вне рамок (НЕ делать в этой задаче)

- L1-кэш эксплореров, bandit-роутинг.
- Изменение самих HITL-гейтов (Asking) и их авто-обхода.
- Ручной relaunch-эндпоинт и UI-кнопка ↻ — оставить как есть (авто-heal их дополняет,
  не заменяет).
