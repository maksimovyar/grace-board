# FIX: verify ненадёжен — «green/ready» прячет реальные дефекты

**Дата:** 2026-07-05
**Где проявилось:** прогон плана из ~13 карточек на проекте `kaizen-planer` (Next.js/TS).
**Симптом:** карточки доезжали до `ready` (verify «зелёный»), но содержали дефекты, которые
всплывали только при ручном ревью диффа и сборке из чистого git-чекаута.

---

## Лог инцидентов (3 за одну сессию)

|---|----------|----------------------|--------------------------|
| 1 | C2 «юнит-тесты replan» | Галлюцинация: переписал **несвязанный** прод-файл `whisper/whisper-server.py` (openai-whisper → faster-whisper). Вне scope карточки | Переписанный файл **внутренне консистентен** — компилируется, импорты резолвятся, тесты зелёные. Ни один механический гейт не проверяет «а это просили?» |
| 2 | C3 «replan через оркестратора» | Реальная ошибка типов `TS2322: unknown → ReactNode` в `AgentChat.tsx:111` | Строгий `tsc --noEmit` не гонялся (либо только `next build`/Turbopack — мягче к части ошибок, либо «зелёный» = самоаттестация агента) |
| 3 | fix-карточка (5 дефектов ревью) | Создал новый файл `lib/replan/constants.ts`, импортит из 3 мест, оставил **untracked** (не в git). В чистом чекауте → `TS2307: Cannot find module` | Файл **физически лежал на диске** в рабочем дереве grace → tsc/build в этом дереве резолвят его |

## Корень: verify проверяет не то

Одна фраза: **grace проверяет «мой диск собирается», а нужно «то, что уходит в git, собирается
и делает ровно то, что просили».** Разрыв между этими двумя — источник всех инцидентов.

Три структурных слепых пятна:

1. **Verify бежит в грязном рабочем дереве, а не в чистом чекауте закоммиченного.**
   Пайплайн коммитит только `files[]` карточки (`git add <files>`, не `git add .` — см.
   `grace-feature-dev/SKILL.md`, green-checkpoint). Но tsc/build при verify видят **все** файлы
   на диске, включая созданные-но-не-добавленные. Отсутствующая в git зависимость невидима:
   на диске резолвится, в git её нет. → инциденты 3 и развал replan.

2. **Verify проверяет «собирается», а не «в scope».** Внутренне непротиворечивая галлюцинация
   проходит все механические гейты. Ловится только сверкой диффа с намерением карточки. → инцидент 1.

3. **«Зелёный» — заявление агента, а не воспроизведённый строгий гейт.** Нет гарантии, что реально
   отработали `tsc --noEmit && <тесты> && build`. И нет инварианта «`files[]` замкнут по импортам»
   (всё, что импортит закоммиченный код, — тоже закоммичено). → инциденты 2, 3.

Плюс поведенческая гоча: grace иногда оставляет правки **незакоммиченными прямо в рабочем дереве
`main`** (без ветки) — карточка `ready`, а работа висит uncommitted.

---

## Как обнаружено

Дефекты поймали **два ручных гейта перед мержем каждой карточки** (ни один не входит в grace-verify):

1. **Ревью диффа против scope карточки** — `git diff main..<branch> --name-status`, скан на файлы
   вне ожидаемого набора. Поймал инцидент 1 (whisper вне scope) и «мёртвый код».
2. **Clean-tree проверка** — свежий `git worktree add --detach <sha>` + симлинк node_modules +
   `npx tsc --noEmit` (+ тесты). Воспроизводит ровно то, что увидят CI/деплой — **только
   закоммиченные файлы**. Поймал инциденты 2 и 3 (untracked-зависимость, ошибка типов).
   Гоча: Turbopack `next build` НЕ принимает симлинк node_modules («symlink points out of
   filesystem root») — build делать в реальном репо, временно убрав untracked из исходников.

## Варианты исправления (в самом grace / grace-board)

Ранжировано по «дёшево × ловит».

### A. Verify — в чистом чекауте, а не в рабочем дереве (главное)
Перед прогоном `tsc/тесты/build` в фазе verify: `git stash -u` (или отдельный `git worktree` на
временном коммите) → прогнать гейты → развернуть обратно. Тогда untracked-файлы не маскируют
отсутствующие зависимости. Закрывает пятно 1 (инциденты 3 + развал replan) — самый частый класс.
Реализация: правка verify-шага в `grace-feature-dev/SKILL.md` (или обёртка в раннере).

### B. Инвариант «files[] замкнут по импортам»
После коммита карточки — проверка, что закоммиченный код не импортит ни одного **untracked**-пути.
Дёшево: `git status --porcelain` по каталогу карточки + греп импортов на untracked-файлы; либо
просто `git add` всех новых файлов в каталоге фичи, а не точечный список. Фейл карточки, если
остались untracked-зависимости. Закрывает пятно 1 на уровне коммита, а не только verify.

### C. Voer — воспроизведённый строгий гейт, а не самоаттестация
Явно фиксировать в SKILL, что verify = **обязательно** `npx tsc --noEmit && <test-runner> run &&
<build>` с ненулевым кодом = провал (карточка → `implementing`, не `ready`). Не полагаться на
«агент посмотрел и решил, что ок». Закрывает пятно 3 (инцидент 2).

### D. Scope-diff гейт (для галлюцинаций)
Автоматический: после реализации — `git diff --name-only` против объявленного `files[]` карточки;
файлы вне списка → блок/эскалация на ревью. Ловит контаминацию (инцидент 1). Полностью механически
scope не проверить (нужно «просили ли»), но «тронул файл вне заявленного набора» — уже сильный сигнал.

### E. Не оставлять незакоммиченным на main
Раннер после прогона должен гарантировать: работа карточки либо на ветке `autodev/<slug>` с
коммитом, либо явный фейл. Не оставлять модификации в рабочем дереве `main` без ветки.

---

## Интерим-митигация (пока grace не пофикшен)

Мержить карточку grace ТОЛЬКО после двух ручных гейтов (см. «Как обнаружено»):
1. ревью диффа против scope;
2. clean-tree `tsc + тесты (+ build)` из свежего worktree/`stash -u`.

Карточка в `ready` сама по себе — НЕ основание для мержа.
