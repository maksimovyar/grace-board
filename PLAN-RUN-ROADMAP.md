# grace-board → «Прогон по плану» (Plan Run)

> Статус: спека + план сборки. Заморожено 2026-07-05.
> Цель: превратить доску из диспетчера одиночных карточек в движок, который
> берёт **одну задачу-цель**, декомпозирует её, выводит на человека дизайн /
> развилки / блокеры / конфликты — все разом, — а затем **последовательно и
> автономно** реализует этапы без потери качества и без гонок по git.

Этот документ — два в одном:
1. **Спека будущих прогонов** (§1–§6.1) — как должен работать автоматический
   прогон и режим AUTO. Это и есть контракт, к которому мы идём.
2. **План сборки** (§7) — этапы, которыми мы этот движок строим (bootstrap,
   вручную, поверх текущей доски).

---

## 1. Сущности

### Plan (прогон)
```
Plan {
  id, goal, project,
  integrationBranch,          // autodev/plan-<id> — все этапы коммитят сюда
  status,                     // planning | designing | clarifying | running | done | blocked
  cardIds[],                  // этапы прогона
  result                      // см. §6
}
```

### Task (этап / карточка) — расширение текущей модели
```
Task {
  ...текущие поля...,
  planId,                     // к какому прогону принадлежит (null = одиночная задача)
  dependsOn[],                // id этапов-предшественников
  autonomy,                   // "auto" | "ask" (см. §5)
  files[],                    // множество трогаемых файлов (порядок очереди по пересечению, §5.2)
  result                      // см. §6
}
```

## 2. Жизненный цикл прогона

```
Фаза 0    Декомпозиция (черновик, редактируемый) + ресёрч: нужен ли дизайн (§3)
Фаза 0.5  Если нужен → дизайн-расширение из источника проекта → АКТУАЛИЗАЦИЯ
          плана (правит ЛЮБОЙ ещё не собранный этап, включая бэк)
Фаза 1    Сводный гейт: **план-уровневые** арх-развилки (дедуп) + блокеры +
          конфликты — решаем разом на обсуждении плана. Гейт НЕ претендует собрать
          все развилки: низкоуровневые, видимые только внутри карточки, агент
          решает сам (порог эскалации, §5.3); класс-пола до-эскалируется по ходу.
          Дизайн уже учтён. Решения прописываются в seed каждой карточки.
Фаза 2    Сборка: DAG-шедулер ведёт этапы; UI-этапы РЕАЛИЗУЮТ макеты; на каждом
          этапе — verify (+ фиделити-verify для UI). Коммиты в интеграционную ветку.
          Каждый этап на `ready` отдаёт свои дельты релиза (§6.1) — они копятся.
Фаза 3    Интеграционная ветка → ОДИН PR в main. Мерж — человек (git-пол, §4).
          К PR приложен МАНИФЕСТ РЕЛИЗА (§6.1) — накопительный итог по 5 разделам.
```

Ключевые инварианты:
- **Дизайн гейтит весь граф сборки** (см. §3) — ни один build-этап не стартует
  до закрытия дизайна+актуализации. Причина: дизайн может тронуть форму данных
  (`plan.json`, поле «Результат»), а не только вёрстку. В `autonomy=auto`
  дизайн-гейт закрывается без человека (генерация + SA), в `ask` — с человеком
  (§3.5); граф ждёт закрытия в обоих режимах.
- **Черновик плана редактируется человеком** до утверждения — страховка от
  плохой декомпозиции.
- **Каждый мокап обязан мапиться на build-этап** (иначе orphan-дизайн);
  соответствие проверяет актуализация (Фаза 0.5).

## 3. Дизайн — правила

### 3.1 Источник дизайна декларируется в проекте
Каждый проект держит указатель на актуальный дизайн в своём `CLAUDE.md`
(или memory). Формат:
```
## Дизайн
Источник: figma | html | none
Ссылка: <figma-url> | <путь к папке, напр. public/>
```
- Тип источника определяет тулинг дизайн-прогона: **figma → Figma MCP**,
  **html → чтение файлов**.
- Строки нет → прогон обязан её завести (спросить/дискаверить и записать) —
  self-healing конвенции.
- Для самого grace-board: `Источник: html · Ссылка: public/`.

### 3.2 Нужен ли дизайн — решают ОБЪЕКТИВНЫЕ сигналы (не «крупная/мелкая»)
Размер задачи субъективен и неизвестен заранее. Опираемся только на текст
задачи + ресёрч:
- **приложен дизайн** (ссылка/файлы) → дизайн = вход декомпозиции, отдельного
  этапа нет, актуализация не нужна (дизайн уже учтён);
- **явное указание сделать дизайн** → дизайн-этап генерится;
- **ни того ни другого** → **ресёрч-прогон решает**: читает задачу +
  задекларированный дизайн проекта (§3.1) + код и выдаёт вердикт
  «бэк-only / нужна доработка UI».

### 3.3 Гранулярность
- дизайн потребляет **один** этап → это **фаза внутри его прогона** (архитектор
  сначала читает существующий дизайн и расширяет, потом кодит) — без отдельной
  карточки;
- дизайн потребляют **несколько** этапов → **отдельный узел** + общий артефакт
  в `design/`, к которому этапы биндятся.

### 3.4 Артефакт передачи «дизайн → код»
Дизайн-этап кладёт в репо (`design/<screen>`): статический мокап + токены +
короткая спека, коммит в интеграционную ветку. Build-этап получает путь к
артефакту **явно в промпте кодера**. Без артефакта «строится по дизайну» —
пустые слова.

### 3.5 Утверждение и верификация — зависит от `autonomy`
Дизайн-гейт ведёт себя по-разному в зависимости от режима прогона:
- **`autonomy=ask`** — дизайн утверждает **человек** (визуал = спека для N этапов),
  плюс цикл итераций до фиксации. Дизайн-этап гейтит граф (§2). Критерий
  готовности этапа — «утверждён человеком», а не «гейты зелёные».
- **`autonomy=auto`** — человека в дизайн-гейте нет: макет генерит **frontend-design
  skill + правила проекта (§3.1)**, сразу за ним SA-актуализация плана, дальше по
  воркфлоу. Критерий готовности — «макет сгенерён и закоммичен в `design/`».
- verify UI-этапа в обоих режимах = функционально **плюс фиделити-чек**
  (скриншот vs мокап из `design/`).

## 4. Git-модель

- **Интеграционная ветка на прогон**: `autodev/plan-<id>`. Все этапы коммитят
  сюда.
- **База этапа = НЕ main.** База = tip интеграционной ветки = `main` (снятый в
  начале прогона) + весь код уже завершённых этапов. Значит этап видит
  результат предшественников.
  - корневые этапы (без deps) — база `main`;
  - каждый следующий — база = tip интеграционной ветки;
  - **этапы идут строго последовательно** (WIP=1, §5.1) и коммитят стопкой на
    одну ветку — параллельных рукавов и `git worktree` НЕТ (см. §5.2). Так внутри
    прогона два этапа никогда не пишут одновременно → intra-run git-конфликтов
    физически не бывает, разруливать нечего.
- **main трогается один раз** — финальный PR в конце прогона (Фаза 3). Мерж
  делает человек.
- Если за долгий прогон кто-то коммитит в `main` — интеграционная ветка
  отстаёт; разбираемся **только на финальном PR** (rebase/merge), а не на
  каждом этапе.

**Текущее состояние (что чиним):** билд уже ответвляет `autodev/<slug>` от
свежего main (`server.js:388`), так что база ветки — не проблема. Реальный
источник гонки: все карточки проекта работают в одном `cwd` (`server.js:270`),
поэтому `git checkout main` одной карточки дёргает рабочий каталог под другой →
параллельный диспатч = риск порчи git. Мерж в main тоже не описан. Закрывается
это **последовательной очередью WIP=1** (§5.1), а не worktree: раз ничего не
параллелим, второго рабочего каталога не нужно.

## 5. Последовательность и AUTO

### 5.1 Очередь (сериализация)
- WIP=1 на проект по умолчанию: выезд из Backlog не диспатчит сразу, а ставит
  в очередь; тик (`server.js:916`) берёт следующую, когда нет активной.
- Заодно чинит гонку общего `cwd`.

### 5.2 Зависимости (DAG)
- `dependsOn[]`: этап диспатчится, только когда все deps в `ready`; чип
  «🔒 ждёт: X».
- Шедулер в тике обходит граф и выдаёт **по одному** готовому этапу за раз
  (WIP=1, §5.1) — **ничего не параллелим**. `files[]` служит для того, чтобы
  этапы с пересечением файлов гарантированно шли друг за другом в очереди;
  worktree и конфликт-детектор рукавов не нужны (см. §4).

### 5.3 Режим AUTO — с жёстким полом
Флаг `autonomy` (глоб. дефолт в шапке доски + оверрайд на карточке). Что меняет:
1. **Порог эскалации в промптах ask** (`server.js:322-325`): при `auto`
   подменяется на «не эскалируй — выбери сильнейший вариант, зафиксируй
   обоснование, продолжай».
2. **Авто-сбор `recommended` в тике** (`server.js:536`): собирает
   `archDecisions` из опций `recommended:true` и запускает build.

**Жёсткий пол (без ухудшения качества):** даже в AUTO определённый класс
развилок ВСЕГДА стопит карточку и идёт к человеку — необратимое, стоимость,
юридика/резидентность/провайдер, форма модели данных, влияющая на контракт.
Автоматизируем обратимое, гейтим необратимое.

**Пол считает НЕ билдер (расщепление роли).** Тот же прогон, которому сказано
«не эскалируй, продолжай», не может честно судить, необратимо ли его решение —
конфликт интересов. Поэтому пол = объединение двух независимых источников:
1. **Механический (без агента):** детерминированный чек по diff + манифесту
   `deploy{}` (§6.1) флагает объективно видимые классы — деструктивная миграция
   (`DROP`/`ALTER` без two-step), новая `*_KEY/_SECRET/_TOKEN` env, удаление
   данных, новый пакет-SDK внешнего API. Флаг есть → стоп к человеку, независимо
   от `autonomy`. Сигналы берём из манифеста, который карточка и так пишет (SR).
2. **Узкий классификатор (отдельный вызов, не билдер):** для судительной части
   (провайдер/юридика/стоимость) — отдельный агент с промптом ровно «ответь
   только: попадает ли решение в {необратимость, стоимость, юридика, форма
   данных в контракте}? да+категория / нет. Ничего не решай». У него нет мотива
   продолжать, поэтому вердикту можно доверять.

Отсюда порядок сборки: **S3 (AUTO) зависит от SR (манифест)** — механический пол
читает `deploy{}`.

**Git-пол:** AUTO **не мержит в main** сам. Автономно гоняем этапы в
интеграционную ветку; мерж в main — единственный человеческий гейт (PR).

**Лог-след:** каждое авто-решение видно на карточке («🤖 авто-решений: N») и в
`dispatch-log` — надзор постфактум, не блокирующий.

## 6. Поле «Результат»

Единый формат на **задаче** и на **прогоне**:
- **git-ссылка** — ветка прогона (интеграционная) / ветка задачи
  (`autodev/<slug>`);
- **арх-развилки, решённые без человека** (из AUTO-следа — `archDecisions`, где
  `ownText` от агента);
- **значимый итог** (`finishNote`);
- при блоке — **ошибка и её причина** (`blockReason`).

Реюз существующих полей: `branchLink` (`server.js:497`), `finishNote`
(`server.js:498`), `blockReason`, `archDecisions`. `result` — агрегат + git-ссылка,
а не новые данные с нуля.

## 6.1 Манифест релиза (5 разделов)

Раскатывается и прогон, и **одиночная задача** — значит манифест нужен обоим.
Это **5 разделов действий на релиз** — что сделать, чтобы раскатать результат:

| Раздел | Что копим |
|---|---|
| **Миграции** | новые Prisma/SQL-миграции (forward-only, backward-compatible); порядок применения |
| **env** | новые/изменённые переменные окружения (+ в `.env.example`), секреты — куда класть |
| **Сервисы/таймеры** | новые/изменённые сервисы, воркеры, systemd-юниты, cron/таймеры |
| **seed/скрипты** | сиды, разовые скрипты миграции данных, backfill — что и в каком порядке гонять |
| **Ручная проверка** | чек-лист прокликать/проверить руками после раскатки |

Контракт (единый для любой единицы — этапа прогона И одиночной задачи):
- **Каждая карточка** на `ready` пишет структурный блок `deploy: { migrations[],
  env[], services[], seed[], manualChecks[] }` в свой `board.json` (обобщение уже
  синкаемого `pip.migration`, `server.js:499`). Это её собственный манифест.
  **Все 5 ключей обязательны**: пустой массив = «проверял, пусто»; отсутствие
  ключа = «забыл» → verify фейлит. Так «правильно пусто» отличимо от «упущено».
- **Одиночная задача** (`planId: null`): `deploy{}` = её манифест; кладётся в
  `task.result.releaseManifest` и **прикладывается к её PR** (ветка
  `autodev/<slug>`). Аккумуляции нет — единица одна.
- **Прогон**: **аккумулирует** дельты этапов в `plan.result.releaseManifest`.
  Слияние — **per-раздел, ключ и порядок под раздел** (единый set-дедуп ломал бы
  порядок миграций):
  | Раздел | Ключ слияния | Порядок |
  |---|---|---|
  | env | по **имени** переменной (конфликт значений → warn) | не важен |
  | migrations, seed/скрипты | по **пути файла / id** (схлопнуть только точный дубль) | **сохранять топологический порядок DAG** = порядок, в котором этапы дошли до `ready`; НЕ сортировать, НЕ дедупить по содержимому |
  | services, manualChecks | по идентичности (имя юнита / текст чека) | first-seen |
  Результат прикладывается к финальному PR интеграционной ветки (Фаза 3).
- Пустой раздел — валиден (бэк-задача без миграций → `migrations: []`).

Это прямое продолжение поля «Результат» (§6): `result` — итог единицы,
`releaseManifest` — тот же 5-раздельный манифест на уровне задачи (свой) или
прогона (накопительный). Готовый ран-бук раскатки, а не «вспоминаем, что менялось».

## 7. План сборки (bootstrap)

Строим движок этапами. **Строго последовательная линейная цепочка** — дизайн
идёт первым и гейтит план, дальше каждый этап опирается на предыдущий. Ничего не
параллелим (§5.2), поэтому `git worktree`-этапа больше нет.

```
SD → SA → S0 → S1 → SR → S3 → S4 → S5   → финальный PR в main
(дизайн) (план) (очередь) (deps) (манифест) (AUTO) (планер) (гейт)
```

| Этап | Что | deps | verify |
|---|---|---|---|
| **SD** | Дизайн-расширение новых поверхностей (черновик плана, сводный гейт, поле «Результат», тумблер AUTO, чипы deps/очередь/блокер/конфликт) через **frontend-design skill + правила проекта** из `public/styles.css` + существующих экранов, в текущем языке; коммит в `design/`. `ask` → утверждает человек; `auto` → генерится без человека (§3.5). | — | ask: человек утвердил макет; auto: макет сгенерён и закоммичен |
| **SA** | Актуализация плана: сверить UI-этапы с макетами; может править ЛЮБОЙ ещё не собранный этап (вкл. форму данных SR/`plan.json`). | SD | соответствие мокап↔этап |
| **S0** | Фундамент: очередь (WIP=1, последовательная) + интеграционная ветка `autodev/plan-<id>` + финальный PR в main; **чинит общий `cwd`** (`server.js:270`) — источник гонки, не база ветки. | SA | 2 карточки проекта — вторая стартует после `ready` первой и видит её коммиты |
| **S1** | `dependsOn[]` + **последовательный** шедулер (WIP=1, без параллели); `files[]` держит пересекающиеся этапы в очереди друг за другом; чип «🔒 ждёт». | S0 | цепочка идёт строго по порядку; пересекающиеся по файлам — не одновременно |
| **SR** | Поле «Результат» (git-ссылка + авто-решения + итог + ошибка/причина) **+ манифест релиза** (§6.1): `deploy{}` со всеми 5 ключами на КАЖДОЙ карточке → `task.result.releaseManifest` у одиночной, per-раздел аккумуляция в `plan.result.releaseManifest` у прогона; приложение к PR. API + вёрстка. | S1 | манифест по 5 разделам заполнен и у одиночной, и у прогона; отсутствие ключа фейлит; виден у PR |
| **S3** | Режим AUTO: флаг, поднятие порога, авто-сбор `recommended`, **расщеплённый пол** (механический по `deploy{}` + узкий классификатор, §5.3), git-пол, лог-след. | SR | без человеческих развилок доходит до `ready`; синтетическая карточка класса-пола стопит |
| **S4** | Планировщик + черновик плана: сущность `Plan`, `launchPlanner`, классификатор дизайна (§3), разрешение источника, гранулярность. Строится по SD. | S1, SA | одна цель → связный набор этапов с корректными `dependsOn` |
| **S5** | Сводный гейт решений: один экран (**план-уровневые** развилки дедуп + блокеры; `files[]` информирует порядок очереди, а не параллель), проброс решений в seed карточек. Строится по SD. | S4, S3 | план-уровневое решение — карточки не переспрашивают |

**Что изменилось против первой редакции:** удалён **S2** (worktree/параллель —
§5.2 их отменяет); порядок линеен вместо параллельной группы `(S0,S3,SR)`, которая
нарушала собственные `dependsOn`; **SR перед S3** и зависимость развёрнута
(`S3 deps=SR`), потому что механический пол читает манифест `deploy{}` (§5.3);
SD-verify раздвоён по `autonomy` (§3.5).

**Порядок перетаскивания:** SD → SA → S0 → S1 → SR → S3 → S4 → S5.

## 8. Как этот план соотносится с будущими прогонами (dogfooding)

Этот план сборки — **ручной прогон ровно того процесса, который мы строим**.
Структурирован он как будущий Plan Run (§2): цель → декомпозиция → дизайн-гейт →
сводный гейт → последовательная сборка → PR. Разница только в исполнителе:

- **Сейчас (bootstrap):** движка `dependsOn`/очереди/AUTO ещё нет — сам его этими
  этапами и делаем. Поэтому первый прогон идёт по текущей механике: `dependsOn`
  выше — **документация порядка**, а фактическую последовательность держим руками
  (тащим по одной, следующую только после `ready` предыдущей). Все этапы правят
  один `server.js` и коммитят стопкой в одну ветку — для чисто последовательной
  модели (§5.2) это норма, а не проблема; именно поэтому S2/параллель и не нужны.
- **По мере готовности этапов enforce включается сам:** после **S0** — очередь
  WIP=1; после **S1** — зависимости; после **S3** — AUTO; после **S5** — полный
  Plan Run из одной цели.
- **После S5** будущие планы (например, этапы DocsInside2 0.x/1.x, kaizen A/B/C
  на доске) запускаются уже автоматическим прогоном: поставил цель → утвердил
  декомпозицию и дизайн → разрулил развилки разом → ушёл → получил PR в main.

То есть текущий план — одновременно и первая фича, и эталон поведения, к
которому AUTO-режим должен сойтись.
